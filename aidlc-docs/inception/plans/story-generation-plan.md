## Questions

### Story Organization

## Question 1
How should the user stories be organized/broken down?

A) Feature-Based — stories grouped by game feature (towers, enemies, controls, UI, economy)

B) User Journey-Based — stories follow the player's experience flow (start game → learn controls → place towers → survive waves → win/lose)

C) Epic-Based — high-level epics (Core Gameplay, Tower System, Enemy System, UI/UX) with sub-stories

D) Other (please describe after [Answer]: tag below)

[Answer]: C

## Question 2
What level of granularity should stories have?

A) Coarse — fewer large stories (8-12 total), each covering a major feature area

B) Medium — moderate number of stories (15-25), each covering a distinct user interaction

C) Fine — many small stories (25+), each covering a single atomic action or behavior

D) Other (please describe after [Answer]: tag below)

[Answer]: A

### Player Persona

## Question 3
What type of player are you primarily designing for?

A) Casual gamer — wants quick fun in the terminal during breaks, values simplicity

B) Strategy enthusiast — enjoys optimizing tower placement and resource management

C) Developer/terminal user — appreciates the novelty of a game in their usual environment, values keyboard efficiency

D) All of the above — design for a broad audience with progressive depth

E) Other (please describe after [Answer]: tag below)

### Acceptance Criteria

## Question 4
How detailed should acceptance criteria be?

A) Given/When/Then format — formal BDD-style scenarios for each story

B) Checklist format — bullet list of conditions that must be true for the story to be complete

D) Other (please describe after [Answer]: tag below)

A) Yes — assign Must Have / Should Have / Could Have (MoSCoW) priorities
